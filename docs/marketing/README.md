# Sable — Marketing & Sales-Enablement Kit

**Audience:** Business development, partnerships, and anyone who takes a Sable pitch outbound — to a prospect, a deck, a cold email, or a discovery call.

**What this is:** The outbound collateral built *on top of* the honest capability inventory in [`../SUITE_CAPABILITIES.md`](../SUITE_CAPABILITIES.md). That doc is the ground truth — what we can actually do and what status each thing is at. This kit turns it into language you can put in front of a prospect **without overselling.**

> **Read [`MESSAGING.md`](MESSAGING.md) before you write or say anything outbound.** It is the load-bearing file in this folder. It defines the measured-vs-interpretive taxonomy, the claims we never make (no "X% growth," no impression guarantees), and approved-vs-avoid phrasing pairs. Everything else here is downstream of those guardrails.

## What Sable is (one paragraph)

Sable is a managed community-growth operation for crypto projects, backed by a proprietary tooling suite. We find under-served, well-funded projects, run an AI diagnostic that grades their community's health and pinpoints exactly where it's fragile, then run the fix: a content factory for short-form video and posts, a reply-amplification engine that hands operators the right reply targets and drafts human-sent replies that read as the operator (never a bot), KOL outreach with warm-intro routing, in-Discord engagement games, and full contribution tracking — then we *prove* the work, with every assisted reply an operator posts rolled into a sanitized "work delivered" record. The client sees a clean health portal; our operators drive it all from a single console.

## Status-label legend (the same convention used across all Sable docs)

| Label | Meaning | Can you promise it outbound? |
|---|---|---|
| ✅ **LIVE** | In production use today (includes internal operator/analyst tools, not only hosted services) | Yes — sell it |
| ✅ **DEPLOYED (dormant)** | Shipped and ready, but inert until an operator triggers it | Yes, framed as "available, operator-initiated" — never "always-on autonomous" |
| 🟡 **IN-TRIAL** | Built and running, scoped to one client / proving out | Yes, framed as "we're piloting this with [client]" |
| 🟡 **IN-DEVELOPMENT** | Built end-to-end but not yet running for any client (e.g. NULO ships dormant) | Pitch as **coming**, never as a running service |
| 🔵 **ROADMAP** | Specified or scaffolded, not yet running | No — pitch as "coming," never as available |

**The one rule that overrides everything:** if you are not sure a capability is LIVE, check [`../SUITE_CAPABILITIES.md`](../SUITE_CAPABILITIES.md). If it isn't labeled ✅, do not present it as something we do today.

## The honesty discipline, in one line

We sell **measured work and measured amplification** — never guaranteed outcomes. Replies are *suggested and human-sent*. Reach/impressions are not a promise (they're unreliable retroactively). Coverage hours and communities-covered are *self-reported context*, not hard metrics. NULO is *in development*, not live. When in doubt, under-claim — the diagnostic and the proof packet do the selling.
